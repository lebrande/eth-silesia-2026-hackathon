# Pages and Routing Implementation Plan

> Ticket: [`thoughts/tickets/fsn_0004-pages.md`](../tickets/fsn_0004-pages.md)
> Related (in-flight): [`thoughts/plans/2026-04-18-FSN-0003-branding.md`](./2026-04-18-FSN-0003-branding.md)

## Overview

Turn `/` into a real marketing landing page (hero + features + how-it-works + footer, placeholder copy, CTA → chat) and move the customer-facing chat from a floating widget into a dedicated `/agent` page. Both pages are public. Backoffice (`/app/*`) is untouched.

## Current State Analysis

- `apps/main/src/app/page.tsx` is an 18-line placeholder: `Navbar` + centered `BRAND.fullName` hero + floating `<ChatWidget />`. No real marketing structure.
- `apps/main/src/components/ChatWidget.client.tsx` is the only customer-facing chat UI — a floating bubble rendered only on `/`. Owns all chat state (`messages`, `uid`, `threadId`), calls `sendChatMessageAction`, embeds `ReadAloudButton`. Working tree (unstaged, FSN-0003) already migrated its color classes to `bg-primary` / `text-primary-foreground` / `focus:border-primary`.
- `apps/main/src/components/ReadAloudButton.client.tsx` is **also** imported by `apps/main/src/app/(authenticated)/app/sessions-table.tsx` — keep it when deleting `ChatWidget`.
- `apps/main/src/proxy.ts:3` gates auth via `PUBLIC_PATHS = ["/", "/login"]`. Anything not listed redirects unauthenticated users to `/login`. A new public page must be added there.
- `apps/main/src/proxy.ts:14` matcher was already widened (in-flight FSN-0003) to exclude `api | _next/* | favicon.ico | icon | apple-icon | opengraph-image`. `/agent` is not excluded, so auth check runs — adding it to `PUBLIC_PATHS` is sufficient.
- `BRAND` (from `@/branding/config`) provides `name`, `fullName`, `tagline`, `seo.description`. Reuse those; extra marketing prose is inline Polish placeholder strings (YAGNI — not adding a marketing copy dictionary).
- `globals.css` tokens (post-FSN-0003): `--color-primary: #e2007a`, `--color-primary-foreground: #ffffff`, `--color-muted: oklch(0.96 0.006 260)`, `--color-border`, `--color-card`, `--color-foreground`, `--color-muted-foreground`. `card-shadow` utility available. `scrollbar-thin` utility available (`globals.css:63`).
- `Navbar.tsx` (post-FSN-0003) is the standard public header: brand link → `/`, "Zaloguj" → `/login`. Safe to reuse on `/agent`.
- `Navbar.tsx:11` "Zaloguj" already points at `/login`; the hero needs its **own** primary CTA pointing at `/agent`.
- `sendChatMessageAction` (`lib/actions/chat.action.ts:25`) is already callable from any client component; no server coupling to `ChatWidget`. Chat logic moves cleanly.

### Key Discoveries

- The widget's chat logic is entirely local (3 refs + 4 `useState`). Moving it to a new `chat.client.tsx` is a copy-paste + drop the floating wrapper and the `open`/`setOpen` toggle. No hook extraction — single consumer, YAGNI.
- `/agent` can reuse `<Navbar />` for the brand header. One component, consistent UX across public pages. The chat lays out beneath it as `flex-1 overflow-y-auto`, composer pinned at bottom.
- Next.js app-router: `app/agent/page.tsx` (server component, dumb wrapper) + `app/agent/chat.client.tsx` (client component, interactive chat) — mirrors how other routes in this repo co-locate client islands.
- Landing is a static server component — no DB, no `sendChatMessageAction` — so Next will prerender it.
- No file other than `page.tsx:2,15` imports `ChatWidget` — safe to delete after move.

## What We're NOT Doing

- No changes to `/app/*` backoffice routes.
- No changes to `/login` or auth flow.
- No new `BRAND` config keys for marketing copy — placeholder Polish strings live inline in `page.tsx`.
- No hero images, illustrations, or video — typography + color only for v1.
- No i18n — Polish only, matches the rest of the app.
- No analytics / tracking events.
- No animations (Framer Motion, etc.).
- No SEO sitemap / structured data — metadata already comes from `BRAND` via the root layout.
- No shared `useChatSession` hook — single consumer, YAGNI.
- Not removing `ReadAloudButton.client.tsx` from the new chat page (keep TTS parity).
- No loading / streaming UI improvements beyond the typing-dots pattern the old widget had.

## Implementation Approach

Build `/agent` first so the landing CTA has a real destination; then rewrite `/` to link to it; then delete the old floating widget. This order keeps the app working after every phase (no broken-link window, and the widget remains a fallback on `/` during Phases 1–2).

#### 2. New file: `apps/main/src/app/agent/chat.client.tsx`

```typescript
"use client";

import { useState, useRef, useEffect, FormEvent } from "react";
import { sendChatMessageAction } from "@/lib/actions/chat.action";
import { ReadAloudButton } from "@/components/ReadAloudButton.client";
import { Navbar } from "@/components/Navbar";

type Message = { role: "user" | "bot"; content: string };

const WELCOME: Message = {
  role: "bot",
  content: "Cześć! W czym mogę pomóc?",
};

const ERROR_MSG = "Przepraszamy, wystąpił błąd. Spróbuj ponownie za chwilę.";

export function ChatPage() {
  const [messages, setMessages] = useState<Message[]>([WELCOME]);
  const [input, setInput] = useState("");
  const [sending, setSending] = useState(false);
  const uidRef = useRef<string | undefined>(undefined);
  const threadIdRef = useRef<string | undefined>(undefined);
  const listRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (listRef.current) {
      listRef.current.scrollTop = listRef.current.scrollHeight;
    }
  }, [messages]);

  async function handleSubmit(e: FormEvent) {
    e.preventDefault();
    const text = input.trim();
    if (!text || sending) return;

    setMessages((prev) => [...prev, { role: "user", content: text }]);
    setInput("");
    setSending(true);

    try {
      const res = await sendChatMessageAction({
        message: text,
        uid: uidRef.current,
        threadId: threadIdRef.current,
      });
      uidRef.current = res.uid;
      threadIdRef.current = res.threadId;
      setMessages((prev) => [...prev, { role: "bot", content: res.message }]);
    } catch (err) {
      console.error("[ChatPage] send failed:", err);
      setMessages((prev) => [...prev, { role: "bot", content: ERROR_MSG }]);
    } finally {
      setSending(false);
    }
  }

  return (
    <div className="flex h-dvh flex-col bg-background">
      <Navbar />

      <div
        ref={listRef}
        className="flex-1 overflow-y-auto scrollbar-thin px-4 py-6"
      >
        <div className="mx-auto flex max-w-2xl flex-col gap-3">
          {messages.map((m, i) => (
            <div
              key={i}
              className={`max-w-[80%] rounded-lg px-3 py-2 text-sm whitespace-pre-wrap ${
                m.role === "user"
                  ? "ml-auto bg-primary text-primary-foreground"
                  : "bg-muted text-foreground"
              }`}
            >
              <div className="flex items-start gap-2">
                <span className="flex-1">{m.content}</span>
                <ReadAloudButton text={m.content} className="mt-0.5" />
              </div>
            </div>
          ))}
          {sending && (
            <div className="max-w-[80%] rounded-lg bg-muted px-3 py-2 text-sm text-muted-foreground">
              …
            </div>
          )}
        </div>
      </div>

      <form
        onSubmit={handleSubmit}
        className="border-t border-border bg-white px-4 py-3"
      >
        <div className="mx-auto flex max-w-2xl gap-2">
          <input
            type="text"
            value={input}
            onChange={(e) => setInput(e.target.value)}
            placeholder="Napisz wiadomość..."
            disabled={sending}
            className="flex-1 rounded-md border border-border px-3 py-2 text-sm focus:border-primary focus:outline-none disabled:opacity-60"
          />
          <button
            type="submit"
            className="rounded-md bg-primary px-4 py-2 text-sm font-medium text-primary-foreground hover:bg-primary/90 disabled:opacity-50"
            disabled={!input.trim() || sending}
          >
            Wyślij
          </button>
        </div>
      </form>
    </div>
  );
}
```

## Performance Considerations

- `/agent` is a client page (interactive chat). Initial payload is ~the size of the old widget plus Navbar — negligible.
- Landing page is fully static (server component with static children, only `new Date().getFullYear()` is dynamic but evaluated at request time in a server component — fine). Next.js will prerender it; sub-100ms TTFB expected.

## Migration Notes

- No DB changes, no env changes, no Railway config changes.
- No URL that existed before disappears: `/` still exists (new content), `/login` and `/app/*` untouched. `/agent` is purely additive.
- Any user with the old `/` bookmarked still lands on a working page.
